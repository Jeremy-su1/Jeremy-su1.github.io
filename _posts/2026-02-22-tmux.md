# Linux 시스템 운영에서 tmux를 사용해야되는 이유

Linux 기반 엔터프라이즈 환경(DB, WEB, WAS, HA Cluster, Kubernetes Node)에서 운영자의 리스크 상당수는 세션 단절(Session Drop)에서 시작됩니다.  
운영자는 단순히 SSH로 접속하는 것이 아니라, 지속 가능한 작업 환경을 관리해야 합니다. 이를 단순히 SSH 중심의 운영 방식은 구조적으로 취약합니다. 최근에는 보안의 강화를 위해 Access Manager, EAMS, Hi-ware, CyberArk등 별도 bastion을 두고 세션 통제를 많이 하고 있습니다.  
이런 관점에런 tmux (Terminal Multiplexer)는 편의 도구가 아닌, 운영 표준 도구로 정의할 수 있을 것으로 보여집니다.  

# Linux 운영 아키텍처에서의 tmux위치
## 전통적인 운영 구조 (tmux미사용)
```code
    [ 운영자 Labtop ]
           |
          ssh
           |
 [jump Host or something] // Access Manager, EAMS, Hi-ware, CyberArk등
           |
          ssh
           |
 [Target Linux Server]
```
위와 같은 구조에서는 아래와 같은 문제점이 있습니다.  
1) SSH Drop -> Process 종료  
2) 다중 모니터링 불가  
3) 작업 단위 분리 어려움  

이 중에서도 운영자들이 겪는 가장 큰 문제점은 다중 모니터링이 불가한 점이 아닐까 싶습니다. 2개의 로그를 timeline 순으로 분석하기 위해서는 같은 세션 창을 하나 더 열어서 모니터 분할을 하면서 사용을 해야되는 현실이 운영을 괴롭게 만듭니다.   

## tmux 기반 운영 구조
```code
       [ 운영자 Laptop ]
              │
             SSH
              │
        [ Jump Host ]
              │
             SSH
              │
       [ Linux Server ]
              │
 ┌────────────────────────────┐
 │        tmux server         │
 │                            │
 │  Session: DR-Test          │
 │    ├─ Window: Cluster      │
 │    ├─ Window: DB           │
 │    ├─ Window: Network      │
 │                            │
 │  Session: Backup-Check     │
 │  Session: Patch-Work       │
 └────────────────────────────┘
```
위와 같이 SSH는 단순 접속 경로만 될 뿐이고, "tmux"는 운영 작업 환경을 만들어 주는 것입니다.
이러한 작업은 SSH와 독립적으로 분리된다는 부분에 있어 핵심이라고 볼 수 있습니다.

# tmux 기본 개념
tmux는 3단계 구조입니다.  
```code
Session > Window > Pane
```
1) Session  
 Session은 하나의 작업 단위를 의미하며, 앞으로 볼 HA 테스트 시나리오에서는 "HA-TEST"가 session이고, 이것이 곧 "작업 단위 프로젝트"를 의미합니다.  

2) Windows   
 논리적 작업 그룹을 의미하며, 한 화면 단위를 말합니다. 이는 여러개 생성이 가능하며, Cluster시나리오에서는 Cluster window,System window, Application window를 의미합니다. "Ctrl +b -> n"으로 Window간 이동을 하게됩니다.   

3) Pane   
 Window 내부를 분할하는 영역을 의미하며, 동시에 서로 다른 명령어 실행이 가능합니다. 상하 또는 좌우 분할이 가능합니다.

# 실 운영 시나리오 기반 설계
## HA Failover 테스트 시나리오
### 테스트 환경
```plan text
 ┌──────────────────────────┐  
 │     Test Environment     │  
 │ Linux                    │  
 │ Pacemaker + Corosync     │  
 │ DRBD or SAN              │  
 │ DB (HANA or PostgreSQL)  │    
 └──────────────────────────┘    
```
### 테스트 목적
* Primary -> Secondary Failover 정상 동작 검증
* Promote/Demote Timing 확인
* CPU / Runqueue 영향도 분석
* 로그 상관 관계 분석

1) 구조 개념
```plain text
Session: HA-Test

Window 1: Cluster
  Pane 1 → pcs status --full
  Pane 2 → journalctl

Window 2: System
  Pane 1 → sar
  Pane 2 → vmstat

Window 3: DB
  Pane 1 → DB log tail
```
이 구조는 다음 3계층을 의미한다.   
a) Cluster layer  
b) OS resource layer  
c) DB/Application layer    

2) Step-by-Step   
2-1) 세션 생성

위와 같이 하나의 세션으로 접속해서 여러가지 창을 바로바로 띄워 사용이 가능하다는 장점이 있다.
아래 표는 위와 같이 분리를 위한 다양한 옵션들을 담고 있다.

1) Session 관련  


|    기능   	|        키 입력 / 명령        	|           설명           	|
|:---------:	|:----------------------------:	|:------------------------:	|
| 세션 생성 	| tmux new -s HA-Test          	| HA 테스트 전용 세션 생성 	|
| 세션 목록 	| tmux ls                      	| 현재 존재하는 세션 확인  	|
| 세션 접속 	| tmux attach -t HA-Test       	| 기존 세션 재접속         	|
| 세션 분리 	| Ctrl + b → d                 	| 세션을 백그라운드로 분리 	|
| 세션 종료 	| tmux kill-session -t HA-Test 	| 세션 완전 종료           	|

2) Window 관련

| 기능           | 키 입력            | 설명               |
| ------------ | --------------- | ---------------- |
| 새 Window 생성  | `Ctrl + b → c`  | 새 논리 작업 창 생성     |
| Window 이름 변경 | `Ctrl + b → ,`  | 현재 Window 이름 변경  |
| 다음 Window 이동 | `Ctrl + b → n`  | 다음 창 이동          |
| 이전 Window 이동 | `Ctrl + b → p`  | 이전 창 이동          |
| Window 번호 이동 | `Ctrl + b → 숫자` | 특정 번호 창으로 이동     |
| Window 목록 표시 | `Ctrl + b → w`  | 전체 Window 리스트 표시 |
| Window 종료    | `Ctrl + b → &`  | 현재 Window 삭제     |

3) Pane 관련

🔹 Pane 분할

| 기능    | 키 입력           | 설명              |
| ----- | -------------- | --------------- |
| 좌우 분할 | `Ctrl + b → %` | 현재 Pane을 좌우로 분할 |
| 상하 분할 | `Ctrl + b → "` | 현재 Pane을 상하로 분할 |


🔹 Pane 이동

| 기능          | 키 입력           | 설명                    |
| ----------- | -------------- | --------------------- |
| 왼쪽 Pane 이동  | `Ctrl + b → ←` | 좌측 Pane 이동            |
| 오른쪽 Pane 이동 | `Ctrl + b → →` | 우측 Pane 이동            |
| 위 Pane 이동   | `Ctrl + b → ↑` | 상단 Pane 이동            |
| 아래 Pane 이동  | `Ctrl + b → ↓` | 하단 Pane 이동            |
| Pane 순환 이동  | `Ctrl + b → o` | 순서대로 이동               |
| Pane 번호 표시  | `Ctrl + b → q` | Pane 번호 표시 후 숫자 입력 이동 |

🔹 Pane 크기 조절

| 기능    | 키 입력                 | 설명             |
| ----- | -------------------- | -------------- |
| 좌측 확장 | `Ctrl + b → Alt + ←` | 현재 Pane 좌측 확장  |
| 우측 확장 | `Ctrl + b → Alt + →` | 현재 Pane 우측 확장  |
| 균등 분할 | `Ctrl + b → =`       | Pane 크기 균등 재배치 |

🔹 Pane 종료


| 기능         | 키 입력                 |
| ---------- | -------------------- |
| 현재 Pane 종료 | `exit` 또는 `Ctrl + d` |


위와 같이 하나의 테스트 시나리오를 수행하기 위해서는 적게는 2개 많게는 6개의 세션을 일일히 열어야되고, 보안에 의해 아무런 입력없이 1800s가 지나고 나면 다시 세션을 열어야되는 상당히 번거롭고, 운영성을 저하하는 일들을 반복하고 있습니다. 이런 부분들을 위와 같은 Software를 사용한다면, 세션이 닫힌다고 해도 다시 들어가면 상태 그대로를 저장하고 있을테니, 운영함에 있어 아주 좋은 Tool이라고 생각됩니다.
